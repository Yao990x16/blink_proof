# BlinkProof

**Don't trust -- Blink.**

BlinkProof 是一个为 Solana Frontier Hackathon 2026 打造的去中心化内容溯源协议。它针对的是生成式 AI 带来的核心真实性问题：图像与短内容如今可以被高速生成、修改和传播，而普通用户几乎无法在同样快的速度下确认其来源是否可信。

该协议将两项 Solana 原生能力结合在一起：

- **State Compression**：以极低成本锚定大规模的感知哈希（pHash）证明。
- **Blinks**：把“验证内容真实性”从繁琐的钱包流程，变成社交场景中的一次直接交互。

目标很直接：如果创作者、媒体机构或平台想证明一份内容是真实的，他们应当可以把证明发布到 Solana 上，并让任何用户通过一次 blink 完成验证。

## 要解决的问题

AI 生成和 AI 编辑内容已经让传统的信任信号失效。

- 被转发的图片不再稳定携带可依赖的源头元数据。
- 水印很容易被裁切、移除或伪造。
- 中心化溯源数据库会带来平台锁定和单点故障问题。
- 现有验证流程太慢，不适合社交媒体这种“几秒内做出信任判断”的场景。

BlinkProof 面向的就是这样一个时刻：用户在信息流里看到一张图时，会立刻问自己：*这是真的吗？是谁证明的？什么时候证明的？*

## 协议愿景

BlinkProof 并不试图把媒体文件本身存到链上，而是存储**紧凑的可验证证明**，并允许外部独立重算。

高层流程如下：

1. 创作者或验证方从媒体内容中生成感知哈希（pHash）。
2. 协议将该指纹与溯源元数据打包。
3. 证明被提交到 Solana 上的压缩状态结构中。
4. Blink 将验证动作直接暴露在社交分发界面上。
5. 用户在信任内容前，可以查看证明、签发者、时间戳与证明路径。

这种设计在保持公开可验证性的同时，把链上成本压低到了可扩展的范围。

## 为什么选择 Solana

BlinkProof 从一开始就是 Solana-first。

- **State Compression** 让我们能够以可接受的成本写入海量证明，而不是把每条证明都变成昂贵的独立账户。
- **Blinks** 把“我在社交媒体上看到内容”与“我现在就验证它”之间的交互距离压缩到最短。
- **高吞吐、低费用、快速终局** 更适合内容验证这种高频场景，而不是只适合低频结算。
- **程序可组合性** 允许后续扩展到信誉系统、委托签发、媒体注册表和索引服务。

## 核心架构

### 1. 媒体指纹

BlinkProof 计划以感知哈希作为核心媒体指纹，而不是直接使用原始文件字节哈希。

- 对缩放、轻度编辑等变体更鲁棒。
- 更贴近社交内容真实性验证的实际需求。
- 数据足够紧凑，适合压缩证明流程。

pHash 预计在链下生成。链上程序负责验证结构、权限和包含关系，而不是执行昂贵的图像处理。

### 2. 压缩证明

每条证明的目标形态是一个压缩叶子，而不是单独的账户。

计划中的叶子字段包括：

- 媒体 pHash
- 签发者 / 证明者公钥
- 内容主体或创作者引用
- 时间戳或 slot 上下文
- 可选的内容 URI 或外部元数据指针
- 可选的 schema / version 字段

Phase 1 预计会基于 `spl-account-compression` 来实现。

### 3. 通过 Blinks 完成验证

Blink 应该允许用户直接在帖子、消息或嵌入场景里验证内容。

预期流程如下：

1. 用户在社交界面点击 Blink。
2. Blink 解析候选媒体指纹与证明引用。
3. 客户端拉取压缩证明上下文。
4. 用户得到明确结果：
   - 已验证
   - 未验证
   - 存在冲突证明
   - 缺失溯源信息

这个协议的核心判断是：如果验证本身不够顺滑，溯源就不会真正被使用。

## 项目原则

BlinkProof 的核心产品判断很简单：

> 内容真实性的验证，必须快到可以嵌入社交点击。

换句话说：

> **Don't trust -- Blink.**
